# Deep Review - 深度代码审查

专业级代码审查工具，整合静态分析、安全审计、架构评估和规范检查。

## 审查范围

| 维度 | 检查项 | 适用语言 |
|------|--------|---------|
| **代码质量** | 设计模式、复杂度、可读性、DRY/SOLID | 全语言 |
| **架构分析** | 模块依赖、耦合度、分层架构、循环依赖 | 全语言 |
| **安全审计** | OWASP Top 10、注入、XSS、CSRF、认证授权 | Web/后端 |
| **内存安全** | 缓冲区溢出、use-after-free、泄漏、空指针 | C/C++ |
| **MISRA C** | 汽车软件编码规范、安全关键规则 | C/C++ |
| **并发安全** | 数据竞争、死锁、线程安全 | 全语言 |

## 审查工作流

### 阶段 1: 初始化

```
审查任务: [填写审查目标]
审查范围: [文件/模块/全仓库]
审查标准: [安全优先/架构优先/规范优先/综合]
```

**收集上下文**:
1. 读取项目架构文档 (AGENTS.md, README.md)
2. 识别技术栈和关键依赖
3. 确定适用的规范标准 (MISRA/AUTOSAR/OWASP)

### 阶段 2: 静态分析

**运行工具扫描** (如可用):
```bash
# C/C++
cppcheck --enable=all --inconclusive src/
clang-tidy -checks='misc-*,bugprone-*,performance-*' src/**/*.cpp

# TypeScript/JavaScript
npm run lint
npx tsc --noEmit

# Python
pylint src/
bandit -r src/  # 安全扫描

# Java
mvn checkstyle:check
mvn spotbugs:check

# Go
golangci-lint run
go vet ./...
```

### 阶段 3: 深度审查

#### 3.1 代码质量检查

**复杂度指标**:
- 圈复杂度 (Cyclomatic Complexity) > 10 需警告
- 认知复杂度 (Cognitive Complexity) > 15 需重构
- 函数行数 > 50 需拆分
- 嵌套深度 > 4 需简化

**设计模式验证**:
- 是否正确应用设计模式？
- 是否存在反模式 (God Object、Spaghetti Code)？
- 依赖注入是否合理？
- 接口隔离原则 (ISP) 是否遵守？

**SOLID 原则**:
- [ ] 单一职责原则 (SRP)
- [ ] 开闭原则 (OCP)
- [ ] 里氏替换原则 (LSP)
- [ ] 接口隔离原则 (ISP)
- [ ] 依赖倒置原则 (DIP)

#### 3.2 架构分析

**模块依赖**:
```
检查项:
1. 是否存在循环依赖？
2. 依赖方向是否符合分层架构？
3. 核心业务逻辑是否独立于框架？
4. 公共库是否过度耦合？
```

**分层架构验证** (适用于 AutoSAR/分层系统):
- MCAL 层是否仅访问硬件寄存器？
- ECUAL 层是否通过 MCAL 访问硬件？
- Service 层是否独立于硬件？
- RTE 层是否正确隔离 SWC？

#### 3.3 安全审计

**OWASP Top 10 检查**:

| 风险 | 检查点 | 严重性 |
|------|--------|--------|
| **注入** | SQL/命令/LDAP 注入防护 | CRITICAL |
| **认证失效** | 弱密码、会话固定、JWT 验证 | HIGH |
| **敏感数据暴露** | 加密、哈希、TLS 使用 | HIGH |
| **XXE** | XML 解析器配置 | MEDIUM |
| **越权访问** | IDOR、权限校验 | CRITICAL |
| **安全配置错误** | 默认密码、调试模式 | MEDIUM |
| **XSS** | 输出编码、CSP 头 | HIGH |
| **反序列化** | 不安全的反序列化 | CRITICAL |
| **组件漏洞** | 依赖库 CVE 扫描 | HIGH |
| **日志不足** | 审计日志、异常记录 | MEDIUM |

**输入验证清单**:
- [ ] 所有外部输入是否校验？
- [ ] 是否使用参数化查询？
- [ ] 文件上传是否验证类型/大小？
- [ ] API 速率限制是否配置？

#### 3.4 内存安全 (C/C++)

**关键漏洞模式**:

```c
// ❌ 缓冲区溢出
char buf[10];
strcpy(buf, user_input);  // 无边界检查

// ✅ 安全写法
char buf[10];
strncpy(buf, user_input, sizeof(buf) - 1);
buf[sizeof(buf) - 1] = '\0';
```

**检查清单**:
- [ ] 缓冲区边界检查
- [ ] 使用后释放 (use-after-free)
- [ ] 双重释放 (double free)
- [ ] 内存泄漏
- [ ] 空指针解引用
- [ ] 未初始化变量
- [ ] 整数溢出
- [ ] 格式字符串漏洞

#### 3.5 MISRA C 合规性

**关键规则** (AutoSAR 汽车软件):

| 规则 ID | 描述 | 类别 |
|---------|------|------|
| **Rule 5.1** | 标识符外部链接唯一性 | Required |
| **Rule 8.4** | 兼容类型声明 | Required |
| **Rule 10.1** | 操作数类型安全 | Required |
| **Rule 11.1** | 指针转换限制 | Required |
| **Rule 17.2** | 递归禁止 | Required |
| **Rule 21.1** | 标准库使用限制 | Required |
| **Rule 22.1** | 资源泄漏检测 | Required |

**MISRA 检查清单**:
- [ ] 禁止使用 `goto` (Rule 15.1)
- [ ] 禁止递归函数 (Rule 17.2)
- [ ] 所有指针使用前验证 (Rule 11.x)
- [ ] 无隐式类型转换 (Rule 10.x)
- [ ] 所有函数有原型声明 (Rule 8.2)
- [ ] 无未使用的变量/参数 (Rule 2.1, 8.1)
- [ ] 循环有终止保证 (Rule 14.x)

## 问题分级标准

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|---------|------|
| **Critical** | 立即利用的安全漏洞/系统崩溃 | 24h | SQL 注入、缓冲区溢出 |
| **High** | 高概率触发的问题/功能错误 | 1 周 | 空指针、数据竞争 |
| **Medium** | 影响可维护性/性能 | 1 月 | 高复杂度、代码重复 |
| **Low** | 风格问题/最佳实践建议 | 下次重构 | 命名不规范 |

## 注意事项

1. **误报处理**: 工具扫描结果需人工验证，避免误报
2. **上下文理解**: 某些"问题"可能是设计决策，需结合业务场景
3. **优先级排序**: 先修复 Critical/High，再处理 Medium/Low
4. **持续改进**: 将常见问题加入 CI/CD 自动化检查
5. **团队协作**: 审查结果与团队共享，建立知识库
